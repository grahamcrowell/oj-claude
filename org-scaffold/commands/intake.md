# /intake — Org Inbound Task Triage and Routing

**Constraint**: This command MUST NOT write to per-project `.claude/` files
or any path outside this org coordination repo. Routing to a project
backlog is **advisory** — emit the proposed backlog item text and let the
operator carry it to the right project. The org repo has no authority over
project files. This enforces the org/project boundary defined in
`D80-org-coordination.md` § Reference File Scoping.

## Purpose

Capture an inbound request, decide whether it belongs in the org backlog
or in one project's backlog, apply standard OpenJunto triage, and present
a draft backlog item plus a routing recommendation. The operator confirms
before any write occurs.

## Execution

### Step 1 — Capture the request

- If the request text is provided as command arguments, use it.
- Otherwise, ask the user: *"What is the inbound request? (one paragraph
  is fine — include scope, motivation, and any deadline if known)"*.
- Read `BACKLOG.md` (org backlog) and `llms.txt` (project inventory) for
  context before scoping.

### Step 2 — Scope triage

Decide between **org scope** and **project scope** using the rule from
`reference/org-coordination-guide.md` § 3:

- **Org scope** if any of:
  - The request affects 2 or more projects.
  - The request introduces or modifies a shared service, infra constant,
    or API baseline already in `validated-facts-org.md` (or that should
    be).
  - The request is a coordination concern (cross-project schedule, shared
    decision, ADR-worthy choice spanning projects).
- **Project scope** otherwise. If project scope, identify **which**
  project by matching the request against the descriptions in `llms.txt`.
  If ambiguous, ask the user to confirm the target project.

State the scope decision and one-line rationale before proceeding.

### Step 3 — Standard OpenJunto triage

Apply the two-dimensional triage:

**A. Execution model** — score the four criteria (multi-domain, regulatory,
production stability, significant cost). 0–1 = Simple, 2–3 = Moderate,
4 = Complex. Apply mandatory escalations (security/compliance/production-risk
→ Complex).

**B. Stakeholder identification** — always include Product Manager and
Distinguished Engineer. Add domain stakeholders based on signals in the
request (security, data, architecture, operations, analytics, ML, quality,
reliability, business). See `~/.claude/reference/stakeholder-guide.md` for
the full mapping.

Report the triage result: tier + stakeholder list + one-line justification.

### Step 4 — Route

**If org scope**: draft a BACKLOG.md entry in the standard format and
propose appending it to the `## Active` section of this repo's `BACKLOG.md`:

```
### BACK-NNN: [Title]
**Priority**: High | Medium | Low
**Status**: Open
**Created**: YYYY-MM-DD
**Owner**: unassigned
**Description**: [What and why, including triage tier and stakeholders]
```

Choose `BACK-NNN` as the next available identifier (scan existing
`BACK-*` headings and increment). Use today's date for `**Created**`.

**If project scope**: produce the same draft entry text **but do not
write it anywhere**. Present it to the operator with explicit routing
instructions:

```
This request is project-scope for [REPO_NAME].

Carry the following entry to [path-or-URL-from-llms.txt]/.claude/BACKLOG.md:

### BACK-NNN: [Title]
**Priority**: ...
**Status**: Open
**Created**: YYYY-MM-DD
**Owner**: unassigned
**Description**: ...

(BACK-NNN should be the next available identifier in that project's
backlog, not this org repo's numbering.)
```

### Step 5 — Confirm before writing

- For **org-scope** items: show the draft entry and ask
  *"Append this to org BACKLOG.md? (y/n)"*. Only on `y` write the entry
  by appending to the `## Active` section. Do not modify any other part
  of the file.
- For **project-scope** items: show the draft and routing instructions.
  Do not write. Confirm the operator has the information they need.

After writing (or after the operator confirms project routing), emit a
one-line summary: `Routed to [org BACKLOG.md | REPO_NAME]: BACK-NNN — [title]`.

## Output format

The command emits, in order:

1. The captured request (echoed for confirmation).
2. The scope decision with rationale.
3. The triage result (tier + stakeholders + rationale).
4. The draft backlog item.
5. The routing instruction or write confirmation prompt.
6. The final one-line summary after the operator confirms.

## Failure handling

- If `llms.txt` has no projects listed and the request is clearly
  project-scope, ask the operator to populate `llms.txt` first.
- If scope is genuinely ambiguous after analysis, default to project scope
  (the smaller blast radius) and note the ambiguity in the draft
  description so the receiving project can re-route to org scope if
  warranted.
- If the operator declines to confirm an org-scope write, discard the
  draft — do not retain partial state.
