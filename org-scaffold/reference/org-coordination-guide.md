# Org Coordination Guide

Day-to-day operational guidance for working with this org `.claude`
coordination repo. This is not a specification — for the authoritative
model see `D80-org-coordination.md`.

## 2. Updating shared reference

**`validated-facts-org.md`** — append a row when a new shared service is
authenticated, an infra constant is confirmed, or an API baseline is
empirically observed. Each row MUST cite evidence (link, runbook, or test
artifact). Re-validate quarterly; flag stale entries during `/health-check`.

**`llms.txt`** — update whenever a repo joins or leaves the org, or when a
project's `.claude/` entry point moves. Keep one-line descriptions accurate
enough that a new team member can find the right repo without opening each.

**Other org reference files** — additions are welcome when material applies
to 2+ projects. Use the same scope-comment header convention as the
generated files, and add an entry under "Shared Reference" in `llms.txt`.

## 3. Org vs. project scope — decision rule

Derived from `D80-org-coordination.md` § Reference File Scoping.

**Org scope** (belongs in this repo):
- Facts and patterns that apply to **2 or more projects**.
- Authenticated service registries (shared APIs, databases, messaging).
- Cross-project coordination commands.
- Cross-project architecture decisions (ADRs that span boundaries).

**Project scope** (belongs in the project's `.claude/`):
- Session state (`state/session.md`).
- Project-specific backlog items.
- Per-environment configs and per-project artifacts.
- Project-only validated facts (a service only this project uses).

**Tie-breaker**: when in doubt, start project-scope. Promote to org scope
once a second project legitimately needs the same content. Premature
promotion creates coupling; late promotion only costs a move.

## 4. Adding a new project to the org

Checklist when a new repo joins the org:

1. **Create the project `.claude/` scaffold** in the new project repo
   (use the per-project OpenJunto scaffolding flow — not this repo).
2. **Add an entry to `llms.txt`** under `## Projects`: `[REPO_NAME]: [path or URL] — [one-line description]`.
3. **Review the project's reference files** — does any content duplicate
   org-scope material in `validated-facts-org.md`? If yes, leave the org
   copy as the source of truth and have the project reference it.
4. **Consider graduation candidates** — does the new project introduce
   facts or commands that other org projects will also need? If yes,
   promote them to org scope now while the work is fresh.
5. **Add the project to `/health-check`** by virtue of step 2 — the
   command iterates over `llms.txt` automatically.
6. **Open a BACKLOG.md item** if the onboarding creates coordination work
   (e.g., shared auth integration, cross-project schema agreement).

## 5. Health check cadence

- **Routine**: weekly during active coordination, monthly during steady state.
- **Triggered**: before any cross-project release, after onboarding a new
  project, after a major shared-service change.

What stale state looks like:

- `llms.txt` lists a project that no longer exists, or omits a project that
  is in active use.
- `validated-facts-org.md` entries older than 90 days with no re-validation.
- `BACKLOG.md` items marked **In Progress** with no owner or no recent update.
- `state/session.md` accidentally committed (gitignore failure).

When found, the `/health-check` command reports but does not fix — the
operator carries the finding to the right repo or backlog item.
